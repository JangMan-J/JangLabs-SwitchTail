# v0.2 direction — composing the exchange

## The shift

v0.1 made the switchboard **read-only**: it reflects the panes that already
exist (ingest snapshots → directory, deck, seat, triage, call log). v0.2 makes
the switchboard a place you **compose** the exchange — spawn boards and lines
directly from inside the plugin, live.

Sequencing decided by the operator: get the **launch / composition front door**
right first; wire the hosted agent sessions for tighter control **afterward**
(see seed `agent-session-wiring`).

## The interaction model

Live and incremental — exactly Zellij's own feel: press a key and it happens.
No spec authored up front, no builder mode, no pending state, no commit step.
"One pane or a hundred depending which buttons you press."

**Compose verbs** (alongside v0.1's navigate/triage keys):
- add a **line** on the current board
- add a **board**

**Parameterizable increment** — the key refinement:
- a bare hotkey acts in **unit** (1 board, or 1 line)
- a **two-press bind** (Zellij idiom: instruction → value) acts in **quantity**
  — verb, then a count → spawn N

So the grammar is *verb + optional count*. Verb alone = 1; verb + digit = N.
The core tracks "mid-bind, awaiting a value" — the same sub-state shape v0.1
already uses for the `i` patch-through prompt (a key opens a state that consumes
the next input), just lighter.

## Defaulting rules (resolved during the conversation)

The composition leans on layered defaults so a sparse action is always
completed, never rejected:
- **Default agent is `claude`.** Every spawned line is staffed by `claude`
  unless a specific command is requested. Out of the box, zero config →
  composing brings up live `claude` sessions, not bare shells. Bare shell is
  the opt-out.
- A board can carry its own command; lines on it run that. Unspecified boards
  fall back to the default agent.
- (Earlier framing of an up-front "X boards × Y agents" *spec* was superseded
  by the live-composition model — the dimension is the running tally of what
  you've pressed, not a blueprint you author. The defaulting intuition carries
  over: partial intent is always filled by the default.)

## Scope boundaries

- **Working directory (cwd) is OUT OF SCOPE for v0.2.** Lines boot wherever the
  plugin's cwd is; *where* each agent works is a later concern (likely rides in
  with the agent-session-wiring slice, or its own phase). This keeps v0.2 to the
  pure compose-the-shape capability.
- No interactive builder/preview; no saved layouts; no named arrangements.
- Vocabulary stays telephony throughout: **exchange** (the whole switchboard /
  session), **board** (tab), **line** (pane), **seat**, **deck**, **trunk**.
  NOT "fleet" / "squad" / "roster" — there are no defined characters, just a
  dimension the operator builds by hand.

## Architecture fit (why this is a clean GSD phase)

Sits squarely on the existing `HostIntent` seam:
- Composition state (which board is current, mid-bind count, what to spawn) is
  **pure `switchtail-core`** — fully unit-testable, no zellij dep.
- Spawning a line is an `OpenLine` intent fan-out (the intent already exists).
- Spawning a board needs one new intent + one dispatcher arm (open a tab).
- No-kill discipline and minimal-permissions posture unchanged (composition
  only opens; it never closes).

## Deferred to plan-phase (deliberately not decided here)

- Exact compose keys and the digit-bind ergonomics (how the count is entered,
  whether a count applies to boards as well as lines).
- Whether `OpenTerminalsOrPlugins` (already declared) covers opening a new
  board/tab, or a new permission/primitive is needed (verify against vendored
  zellij-tile source — do not assume).
- How the default-agent command is configured (extends today's `line_command`
  load key vs. a richer surface).
