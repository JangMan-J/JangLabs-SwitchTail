<objective>
Close the shared root cause behind BOTH Phase-4 UAT gaps: SwitchTail stores
the operator's cursor as a bare row index (`Exchange.selected: usize`) into a
view (`sorted_lines()` / `log_view_calls()`) whose order is NOT stable. When
the view reorders under the cursor — most violently when `R` rings a line in
`SortMode::RingingFirst` and `refresh_ring_flags()` instantly re-sorts that
line to the top — the index keeps its numeric value and the cursor silently
lands on a different line. Every action (R/a/p/s/m/i/Enter, and the log-view
equivalents) then resolves through the drifted index and mistargets.

This plan re-anchors selection by STABLE IDENTITY: a `LineId` in Directory
view, a call `seq` in Log view. The row index becomes a derived, render-time
quantity. This is a pure-core fix, fully unit-testable, and it directly fixes
the ring-mistargeting gap (UAT test 6) while hardening the swap/say/focus/
answer/park targeting that all share `selected_line()`.

Root-cause evidence: `.planning/debug/ring-targets-wrong-line.md` (RingingFirst
re-sort-under-cursor repro, lines 0+2-ringing live dump reconstruction) and the
cross-link note in `.planning/debug/seat-swap-not-positional.md` (swap resolves
its target through the same drifting index).

Purpose: Selection must mean "this line / this call," not "row N of whatever
the view currently shows." Without this, the positional-swap fix in 04-06 would
inherit the same mistargeting.
Output: An identity-anchored selection model in `switchtail-core`, with tests
that reproduce the exact RingingFirst re-sort-under-cursor scenario and prove
the cursor stays on the operator's line.
</objective>

       `selected_line: Option<LineId>` and `selected_seq: Option<u64>`

       row index is now derived. (If `selected` is read anywhere outside
       exchange.rs, replace those reads with the resolution helpers below —
       grep the workspace for `.selected` first.)

    2. Add resolution helpers:
       - `selected_line()` (already exists): in Directory, return
         `self.selected_line` filtered to a still-present line, falling back to
         the first line if the anchor is gone/empty. In Log, resolve via the
         anchored seq → the call's `line`.
       - `selected_row()` (new, `pub(crate)`): the index of the anchor within
         the CURRENT `sorted_lines()` / `log_view_calls()` ordering, for the
         renderer. Returns `Option<usize>` (None when the view is empty).
       - `selected_call_seq()` (make `pub(crate)` so view/tests can read it):
         return `self.selected_seq` filtered to a still-present call.

    3. Rewrite the navigation handlers so j/k/Up/Down map anchor→current row,
       step the row within bounds, then map the new row back to the anchor:
       - Directory: row = position of `selected_line` in `sorted_lines()`
         (default 0 if unset), clamp `row±1` to `[0, len)`, set
         `selected_line = sorted_lines()[row].id`.
       - Log: same against `log_view_calls()` and `selected_seq`.

    4. On `Tab` (view switch) and `o` (sort change): re-derive/seed the anchor
       for the now-active view to its first row (preserves "selection starts at

       index that no longer exists.

    5. In `ingest_panes`, REMOVE the `clamp_selection()` length-clamp logic and

       anchored line must still resolve sanely.

    6. CRITICAL: do NOT change `refresh_ring_flags()` or `sorted_lines()`

       sort stops happening. (The diagnosis lists "ringing-first should not

    Use exact LineId/Call types already in scope (`crate::line::LineId`,

    `pub(crate)` accessors the view needs.

|----------|-------------|
| operator keys → core | UI key events drive selection mutations (trusted, in-process) |
| pipe ops → core | external agent ring/status payloads reorder the view (untrusted content, but only mutates flags) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-05-01 | Tampering | selection anchor vs re-sortable view | mitigate | Anchor by stable identity so external pipe ring/status re-sorts cannot redirect the operator's pending action to a different line (the reported bug class) |
| T-04-05-02 | Denial of Service | anchor resolution on closed line | mitigate | Anchor resolution falls back to first present line / None — never indexes out of bounds, never panics on sparse-id close |
| T-04-05-03 | Tampering | npm/pip/cargo installs | accept | No new dependencies introduced; pure-core edit within existing crate |
