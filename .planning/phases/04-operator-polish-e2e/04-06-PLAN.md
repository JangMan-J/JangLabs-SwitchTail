<objective>
Close the seat-swap UAT gap (test 4): "the swap should cause both windows to
exchange their positions precisely so that the layout remains the same but the
terminals have traded places."

Root cause (verified at the installed host's exact source commit, zellij-git
0.44.1.r33.ge9173cb — see `.planning/debug/seat-swap-not-positional.md`): the
current implementation dispatches ONE `replace_pane_with_existing_pane(seat,
line, suppress=true)` call, but that primitive is by host construction a
one-way "bring that pane here" (built for pane pickers): the host extracts the
line (its slot collapses and the tab may auto-relayout), places it in the
seat's geometry, and hides the seat pane in `suppressed_panes`. The seat never
lands in the line's old slot. No `swap_panes(a,b)` primitive exists anywhere in
the zellij plugin API (exhaustive PluginCommand scan) — the one-call
implementation was inherently incapable of expressing the designed semantics.

Fix (diagnosis "Suggested Fix Direction", Option A — fully within the five
declared permissions): compose the exchange as three host calls:

1. `open_terminal_pane_in_place_of_pane_id(line, ".", close_replaced_pane=false)`
   — pins the line's slot with a throwaway placeholder P before extraction can
   collapse it (line becomes suppressed but stays pid-addressable; host
   `extract_pane` has a suppressed-pane branch). Returns `Option<PaneId>` = P.
2. `replace_pane_with_existing_pane(seat, line, suppress=true)` — line takes
   the seat's slot; seat pane suppressed (still addressable).
3. `replace_pane_with_existing_pane(P, seat, suppress=false)` — seat takes P's
   slot (= line's original slot) and the plugin-owned placeholder P closes.

Architectural decision (document in intent.rs — this is the one sanctioned
deviation from "one shim call per intent"): core emits a SINGLE composite
intent `SwapPanes { seat, line }` and the adapter's ONE dispatcher arm executes
the fixed 3-call transaction. Per-call intent variants are impossible without
corrupting the seam: step 3 consumes the placeholder `PaneId` returned by step
1, which is HOST-ALLOCATED — core can never know it, and threading it through
per-call intents would force hidden pending-placeholder state into the adapter
(worse than a self-contained transaction arm). The business decision ("exchange
these two lines") stays in core; the 3-call mechanics are host-API plumbing.

Step 3's `suppress=false` close of the SELF-CREATED placeholder is surfaced as
an explicit owner decision (Task 1) versus the no-kill discipline — default
proceed: P is plugin-owned, lives milliseconds, never holds operator work, and
the mechanism is a parameter of `replace_pane_with_existing_pane`, not any
`close_*` shim (the no-kill guard grep does not fire).

Explicitly OPTIONAL and deferred (per gap-closure instruction): the
adjacent-pane shortcut via `move_pane_with_pane_id_in_direction` (a documented
true positional swap, but only with the adjacent pane in a direction). It is an
optimization, not needed for the gap truth, and requires geometry plumbing into
PaneSnapshot + adjacency math in core. Record it in docs and the SUMMARY as a
follow-on; do NOT implement it here.

Purpose: the operator's hot-seat exchange must do what it says — trade places,
layout unchanged — and must be confirmed live (FIFO ordering and the
suppressed-restore edge cannot be asserted headlessly with confidence).
Output: corrected intent + composed adapter transaction + corrected docs +
live human-verified swap and ring targeting.
</objective>
