<objective>
Wire the board verb to the host (COMP-01/02/11 at the adapter layer): add the
`SpawnBoard` dispatcher arm (→ `open_command_pane_in_new_tab`, DISCARDING its
return), declare the `RunCommands` permission so the fan-out's `open_command_pane`
lines succeed, subscribe to `EventType::CommandPaneExited` and route it to the
core's `note_command_exit` (exit-127 surfacing, no-kill), re-seed the headless
permission cache, build the debug wasm, and prove the whole board verb live via a
reload smoke + a daltonized human-verify checkpoint.

This plan is the ONLY one in the phase with a host effect that runs `claude`, the
permission change, and the live verification — so it is last (wave 3), depends on
the key foundation (01-01) and the core spawn machinery (01-02), and is
non-autonomous (it has a blocking human-verify checkpoint).

OWNERSHIP SPLIT (COMP-11): 01-02 owns the CORE `note_command_exit` + its unit
test; THIS plan owns the adapter `subscribe(EventType::CommandPaneExited)` + the
`Event::CommandPaneExited` arm routing into `note_command_exit`, plus the live
proof.

Purpose: the operator presses the verb and a board of claude lines actually
appears — the "press → it happens" feel, proven on the real Zellij event model.
Output: the spawn dispatcher arm, the declared permission, the exit subscription
+ routing, the re-seeded cache, the debug wasm, and a live-verified board spawn.
</objective>

This plan introduces (adapter side):
- `HostIntent::SpawnBoard` dispatcher arm in `main.rs`: builds a `CommandToRun`
  (path + args, verbatim argv — never a shell) and calls
  `open_command_pane_in_new_tab(cmd, BTreeMap::new())`. The returned
  `(Option<usize>, Option<PaneId>)` is DISCARDED — exactly like today's OpenLine
  arm discards its `Option<PaneId>`. Core registers the board + lines via the
  async TabUpdate/PaneUpdate events; the return value is NOT wired back to core.
- `PermissionType::RunCommands` added to the `request_permission` set in `load()`.
- `EventType::CommandPaneExited` added to the `subscribe(...)` list (v0.1
  subscribes PaneUpdate/TabUpdate/Key/PermissionRequestResult/CwdChanged but NOT
  CommandPaneExited — add it) + an `Event::CommandPaneExited(id, status, _)` arm
  routing to `self.exchange.note_command_exit(LineId(id), status)` and dispatching
  any returned (non-kill) intents.
- Re-seeded isolated permission cache in `tests/e2e.sh` including the RunCommands
  grant (so headless runs do not hang on the interactive re-prompt).
</artifacts_this_phase_produces>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| core SpawnBoard/OpenLine intent → adapter → open_command_pane_in_new_tab / open_command_pane | the resolved agent command becomes a host-run process |
| plugin → host permission grant (RunCommands) | a wider permission surface than v0.1 is requested and granted |
| host CommandPaneExited event → adapter → core note_command_exit | host-reported immediate exits cross into the model |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-08 | Elevation of Privilege | RunCommands permission | mitigate | RunCommands is a deliberate, owner-approved (2026-06-13) addition, scoped to running the operator-configured agent command via native command panes. The grant is interactive (the operator approves it); no command runs without the grant. The permission set stays otherwise minimal (no WebAccess, no InterceptInput, no RunActionsAsUser). |
| T-01-09 | Tampering / command injection | CommandToRun argv from agent_command | mitigate | The adapter passes the command as `CommandToRun::new_with_args(path, args)` — verbatim argv, NEVER a shell string — so there is no shell-metacharacter injection surface. The command source is operator config, not remote input. |
| T-01-10 | Destruction of data (no-kill) | SpawnBoard arm + CommandPaneExited routing | mitigate | The new arm adds no close/kill shim; CommandPaneExited only logs via note_command_exit. The no_kill_guard.rs grep test — run inside `tools/dev.sh test` (the authoritative proof; a bare grep is only a convenience) — stays green (verified in Task 1's automated check). |
| T-01-11 | Denial of Service | first-launch permission re-prompt / headless hang | mitigate | tests/e2e.sh pre-seeds the isolated cache with the RunCommands grant so headless runs do not hang; the interactive re-prompt is a one-time operator approval documented in the SUMMARY + checkpoint. |
| T-01-SC | Tampering | npm/pip/cargo installs | mitigate | No new dependencies added; only adapter code + an e2e shell-cache seed change. Package-legitimacy gate N/A (no install task). |
