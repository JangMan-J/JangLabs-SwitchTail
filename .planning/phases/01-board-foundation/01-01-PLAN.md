<objective>
Close the v0.1 key-model gap (COMP-09): the core `KeyInput` is a bare `Char(c)`
and the adapter's `key_input()` REJECTS modified keys, so no compose verb can
bind on Shift/Super. This plan extends the core key vocabulary to carry modifier
information (Shift / Super) and rewires the adapter's key mapping to preserve it,
then adds a single CONFIGURABLE compose-verb binding (char + required modifiers)
on `Exchange`, read from plugin config in `load()`.

This is the FOUNDATIONAL refactor every later compose verb (this phase's board
verb in 01-03, Phase 2's count grammar, Phase 3's line verb) depends on, so it
is sequenced first and alone. It deliberately does NOT spawn anything yet — the
board-spawn behavior lands in 01-03 once the key foundation and the spawn intents
(01-02) exist.

Purpose: make compose verbs bindable on Shift/Super without colliding with
Zellij's Ctrl/Alt space and without breaking any v0.1 key.
Output: a modifier-aware `KeyInput`, a config-driven `compose_board_key` binding,
and an adapter key mapping that no longer drops modified keys.
</objective>

    pub enum BareKey { Char(char), Enter, Esc, Up, Down, Tab, Backspace }
    pub struct KeyInput { pub bare: BareKey, pub shift: bool, pub super_: bool }
    // ergonomic constructors:
    //   KeyInput::ch('c')                  -> { bare: Char('c'), shift:false, super_:false }  (unmodified char)
    //   KeyInput::key(BareKey::Tab)        -> unmodified bare key
    //   KeyInput::new(bare, shift, super_) -> modified key

    // fn KeyInput::matches(&self, b: &KeyBinding) -> bool

- `BareKey` (NEW inner enum in `key.rs`): the seven bare alternatives the old
  flat `enum KeyInput` carried (Char, Enter, Esc, Up, Down, Tab, Backspace).
- `KeyInput` (REVISED in `key.rs`): the struct above — `bare: BareKey` + `shift`
  + `super_` flags — replacing the bare `enum KeyInput`. Public name unchanged
  (re-exported from lib.rs:23). Derives Debug, Clone, Copy, PartialEq, Eq.
- Ergonomic constructors: `KeyInput::ch(c)` (unmodified char), `KeyInput::key(bare)`
  (unmodified bare key), `KeyInput::new(bare, shift, super_)` (modified).
- `KeyBinding` (NEW in `key.rs`): the configured compose-verb binding (char +
  required Shift/Super), with a `KeyInput::matches(&binding)` predicate.
- `Exchange.compose_board_key: KeyBinding` field: the configured compose-verb
  binding, defaulting to a sensible Shift (or Super) + letter combo.

VERIFIED (cite, do not re-litigate): `KeyModifier::Super` EXISTS in

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| operator keyboard → zellij Key event → adapter key_input() → core key() | host-delivered key events, including modifier combos, cross into the model |
| plugin config (KDL) → load() | operator-supplied configuration string crosses into the model as the compose binding |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-01 | Tampering | key_input() modifier mapping | mitigate | Reject Ctrl/Alt-modified keys (return None) so a malformed/extra-modifier event cannot smuggle into a verb match; only Shift/Super are modeled. Unit-tested in Task 1 (matches() extra-modifier non-match). |
| T-01-02 | Denial of Service | config parse of the compose binding in load() | mitigate | Parse must be total — an unparseable/empty binding config falls back to the Exchange default, never panics the plugin on load. |
| T-01-03 | Elevation of Privilege | core/adapter seam | accept | No new permission and no host-effect intent is added in this plan; the key refactor is pure model + dumb mapping. RunCommands surface is deferred to 01-03 where it is threat-modeled. |
| T-01-SC | Tampering | npm/pip/cargo installs | mitigate | No new dependencies added; this plan touches only existing crates. No package-install task, so the legitimacy gate is N/A. |
</threat_model>
