<artifacts_this_phase_produces>
This plan introduces:
- `HostIntent::SpawnBoard { command: Vec<String> }` (in `intent.rs`): spawn a new
  board whose first line runs `command`. Adapter (01-03) maps it to
  `open_command_pane_in_new_tab`. One intent = one shim call (the FIFO-ordered
  fan-out preserves the existing seam contract).
- `Exchange.agent_command: Vec<String>` field (default `["claude"]`): the
  default agent command resolved in core for compose-verb lines. Distinct from
  `line_command` (the `n` bare-shell opt-out), which is unchanged.
- `Exchange.lines_per_board: usize` field (default 5): how many lines a freshly
  spawned board carries.
- The compose-verb board-spawn fan-out in `key()` (wired into the 01-01 compose
  branch): `[SpawnBoard { agent_command }, OpenLine { agent_command }×(lines_per_board-1)]`.
- A deck-cap warning helper: compares the would-be line count against remaining
  `DECK_KEYS` slots and places a CB-safe amber call-log warning on overflow.
- `Exchange::note_command_exit(&mut self, line: LineId, status: Option<i32>) -> Vec<HostIntent>`:
  places a `LineExited` call-log entry (flagging 127 specially) for a known line,
  retaining the line, returning NO close/kill intent. The adapter (01-03) calls
  this from the host `CommandPaneExited(pane_id, Option<i32>, _)` event.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| operator config (agent_command) → core → SpawnBoard/OpenLine intents → host command pane | the configured agent command becomes a process the host runs |
| host CommandPaneExited → note_command_exit | host-reported immediate exits cross into the model |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-04 | Tampering / Elevation | agent_command carried by SpawnBoard/OpenLine | mitigate | Core passes the configured command verbatim as argv (CommandToRun path+args in 01-03), NEVER through a shell — no shell-interpolation/injection surface. The command is operator-controlled config, not remote input, so the trust source is the operator themselves. Document in the SUMMARY that agent_command should be an absolute path or guaranteed-PATH wrapper (the exit-127 PATH risk). |
| T-01-05 | Denial of Service | deck-cap / large fan-out | mitigate | Phase 1 spawns a fixed default (5), within deck capacity and far below any PaneUpdate-flood threshold; the deck-cap warning surfaces overflow without dropping lines. (Phase 2's N-board counts are where a hard fan-out cap would be re-evaluated.) |
| T-01-06 | Destruction of data (no-kill) | note_command_exit | mitigate | An immediately-exited line is LOGGED (LineExited), never closed/killed; note_command_exit returns no close/kill intent and the no_kill_guard.rs test stays green. The operator keeps the exited pane for inspection/re-run. |
| T-01-07 | Repudiation | board-fill reconciliation | accept | Every spawned line is recorded as a LineOpened call-log entry on ingest (existing path); the call log is the audit surface. No additional measure needed. |
| T-01-SC | Tampering | npm/pip/cargo installs | mitigate | No new dependencies added; only existing core crate code changes. Package-legitimacy gate N/A (no install task). |
</threat_model>
